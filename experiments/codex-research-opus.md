# Research: "Neural App Runtime" / Projection-Based Applications

**Date compiled:** 2026-04-30

> **Audit note:** This is a preserved raw model-assisted research artifact. Some phrasing is intentionally left as originally generated. Use `../references/consolidated-research.md` as the canonical synthesis and `../references/source-status.md` for citation hygiene/corrections.

**Focus question:** Where, in the published literature and in shipping systems, does the *neural network's own internal state* serve as the application — with the visible UI being a real-time projection of that state, rather than code generated by the model or a code layer between user and model?

This file collects the closest analogues found across academia, protocol work, systems, and startups, and analyzes the gap between each one and the strict "projection" definition.

---

## 0. The strict definition we are testing against

For something to qualify as a **projection-based application** in the strong sense:

1. **The model's hidden state holds the app's state machine, domain logic, and data.** No external database, no external control flow, no external rule engine.
2. **The visible UI is a *rendering* of that state**, computed each tick by the model itself or by a thin display function applied to model outputs. The UI is a view, not a program.
3. **There is no persistent code layer** sitting between user input and model state. (Tool calls that mutate external data, or HTML that runs JS handlers locally, count as "code layer" leaks.)

Under this strict reading, almost everything below either (a) keeps state in code/tools and uses the model only for display, or (b) keeps the visual layer in the model but loses state on context overflow / restart, or (c) is a video-frame-prediction simulator (a world *model*) without a UI/business-logic layer at all. The closest existing exemplars are **Mirage (hallucinated apps)** and **GameNGen / Oasis / Genie 3 (neural game engines)**, but each falls short in specific ways noted in §4 below.

## 1. Karpathy's framing — Software 2.0 → 3.0 → "people spirits"

### 1.1 Software 2.0 (2017)

- **Source:** Andrej Karpathy, "Software 2.0," Medium, Nov 2017. https://karpathy.medium.com/software-2-0-a64152b37c35
- **Summary:** Establishes the now-canonical claim that "the source code comprises 1) the dataset that defines the desirable behavior and 2) the neural net architecture that gives the rough skeleton of the code, but with many details (the weights) to be filled in." Weights *are* code; training is compilation; the "program" lives in parameter space.
- **Relation to projection:** Foundational. Software 2.0 establishes that the *program* can live in weights. It does **not** address whether the *application's state* (data, in-flight workflow, UI tree) lives there too, nor how it would be rendered. The 2017 essay is about replacing handwritten classifiers with trained networks, not about replacing entire stateful applications.

### 1.2 Software 3.0 / "LLM as kernel" (Karpathy, 2023–2026)

- **Sources:**
  - Karpathy, X/Twitter: "LLMs not as a chatbot, but the kernel process of a new Operating System." https://x.com/karpathy/status/1707437820045062561
  - YC AI Startup School keynote, June 2025: "Software is changing (again)." https://www.latent.space/p/s3
  - AI Ascent 2026 fireside, April 2026 (per Dealroom recap): https://app.dealroom.co/news/note/vibe-coding-was-just-the-warmup-andrej-karpathy-on-the-dawn-of-software-3-0
- **Summary:** Three paradigms — 1.0 explicit code, 2.0 trained nets, 3.0 LLMs programmable in English. LLM is the new kernel; "chat was a great introductory interface… but it's only one UX paradigm and quite low bandwidth." Karpathy's MenuGen anecdote (AI Ascent 2026) — that his app became obsolete the moment Gemini could do the same thing inline via Nanobanana — is the clearest public articulation of the projection thesis: *an app whose entire state machine is the model has nothing to wrap*.
- **Relation to projection:** This is the **closest articulated framing** of projection-based apps in the current discourse. But Karpathy stops short of the strict definition: in the same 2026 talk he advocates "agent-native infrastructure," "partial autonomy," "autonomy sliders," and verification loops — i.e., he still expects code, tools, and external services around the model. The model is the kernel, not the entire OS.
- **Gap:** Karpathy's vocabulary ("kernel," "interpreter," "people spirits") gestures at the projection thesis without ever reifying it as a system architecture. No published Karpathy artifact specifies "the model's KV cache *is* the app state" or "the UI is a learned function of hidden state."

### 1.3 "People spirits" / simulators
- **Source:** Karpathy on X (Dec 2025): "Don't think of LLMs as entities but as simulators." https://x.com/karpathy/status/1997731268969304070
- **Summary:** LLMs are stochastic simulators of people with emergent psychology; query them by asking *whom* to simulate, not what they "think."
- **Relation to projection:** This is the conceptual bridge. If the model is a simulator, then in principle a single inference loop *can* simulate "what this app does next." But Karpathy uses this framing for personalization and reasoning, not for replacing applications.

## 2. Closest existing exemplars — apps where the model genuinely holds the state

### 2.1 Mirage (Among AI, Dec 2025) — *the most direct existing implementation*

- **Source:** "Hallucinate any App, One Screen at a Time," AmongAI, 2025-12-10. https://amongai.com/2025/12/10/hallucinate-any-app-one-screen-at-a-time/
- **Summary:** Toy/POC. User describes the app in a sentence. Claude Haiku 4.5 emits HTML+CSS for the initial screen. Each click/keystroke is sent back to the LLM, which emits the next HTML view "showing the new state of the app." Demonstrated apps: to-do list, Linux terminal, LISP interpreter, Win95 file manager. Author's quote: "Besides the HTML… there is **no code that ever gets generated**. All the behaviors and functionalities of your app are made up by the LLM as you use it."
- **Relation to projection:** **This is the literal projection thesis as a working artifact.** The state machine is fully inside the model — there is no SQL store, no event bus, no JS handlers; the entire "to-do list" exists only as tokens the model has agreed with itself to emit. HTML is the projection layer, played-forward each tick.
- **Gaps:**
  - **No persistence outside context.** When the context fills, the to-do items vanish. State is bounded by the KV cache.
  - **Latency.** Each interaction is a full LLM round-trip; ~seconds per "click."
  - **Reliability.** State can drift mid-session because the model is re-deriving "what the app is" each turn.
  - **HTML still renders client-side**, so there is technically a "code layer" (the browser executes JS embedded in the hallucinated HTML). But there is no *application-specific* code — the strict-projection condition is only weakly violated.
- **This is the best public reference for someone trying to build the strong version.** It's a toy, but it proves the point.

### 2.2 GameNGen (Google, Aug 2024)

- **Source:** "Diffusion Models Are Real-Time Game Engines," arxiv 2408.14837. https://arxiv.org/abs/2408.14837. Project page: https://gamengen.github.io/
- **Summary:** A diffusion model conditioned on past frames + key presses generates DOOM at 20 fps on one TPU. "Complex game state updates such as tallying health and ammo, attacking enemies, damaging objects, opening doors" are performed inside the model — there is no DOOM source code in the loop at inference.
- **Relation to projection:** **Visual analog of the projection thesis.** The neural net is the entire DOOM runtime; pixels are the projection of its hidden state. Importantly, the *game state machine* (HP, ammo, enemy positions) is implicit in the network's activations, not in any external store.
- **Gaps:** It's a video frame predictor, not an application. There is no semantic UI, no mutable user data, no persistence, and no domain logic that anyone would call "an app." It also drifts after a few minutes (the neural sim "forgets" rooms).

### 2.3 Decart Oasis / Oasis 2.0 (Oct 2024 – 2025)

- **Sources:** https://oasis-model.github.io/, https://decart.ai/, TechCrunch coverage. https://techcrunch.com/2024/10/31/decarts-ai-simulates-a-real-time-playable-version-of-minecraft/
- **Summary:** Real-time Minecraft-like world generated frame-by-frame from keyboard/mouse input by a single foundation model. "No physics engine; just a foundation model." 20 fps, autoregressive.
- **Relation to projection:** Same shape as GameNGen — model is the runtime, frames are the projection. Decart explicitly markets it as "engine-free."
- **Gaps:** Same as GameNGen. Quickly forgets level geometry; not a productivity application.

### 2.4 DeepMind Genie 3 (Aug 2025)

- **Source:** https://deepmind.google/blog/genie-3-a-new-frontier-for-world-models/. TechCrunch: https://techcrunch.com/2025/08/05/deepmind-thinks-genie-3-world-model-presents-stepping-stone-towards-agi/
- **Summary:** Text-prompted, real-time interactive 3D worlds at 720p/24 fps. "Promptable world events" let you alter weather, add actors, etc. while the world is running. ~1 minute of consistent memory.
- **Relation to projection:** Closest world-model to a "live application" — the model holds enough state to keep the same building intact for ~1 minute and respond to in-world textual edits.
- **Gaps:** Demo-ware (Ultra subscriber preview). No notion of users, accounts, files, or persistent business logic. Memory window is too short for any productive application.

### 2.5 Sora / Sora 2 (OpenAI)

- **Source:** "Video generation models as world simulators." https://openai.com/index/video-generation-models-as-world-simulators/
- **Summary:** Video generation models exhibit emergent 3D consistency, object permanence, and "digital world simulation" (controlling a Minecraft player while rendering the world).
- **Relation to projection:** OpenAI's framing — *video models as world simulators* — is structurally identical to the projection thesis: the model maintains a coherent world inside its weights and renders frames as projections. The Minecraft demo inside Sora is particularly suggestive: the player-controllable world is, transiently, an app.
- **Gaps:** Not interactive in real time (Sora is generation-time, not playback-time). And like Genie 3, no app-layer concepts.

### 2.6 Websim.ai

- **Sources:** https://websim.com/. Coverage: https://uxdesign.cc/youre-not-hallucinating-simulated-ais-are-the-future-of-internet-business-e3351b7145cb
- **Summary:** Type a fictitious URL → get an interactive hallucinated website. LLM (Claude 3.5 Sonnet by default) interprets the URL as a prompt and generates the page; clicks generate the next page.
- **Relation to projection:** Closer to Mirage than to GameNGen — the "app" exists only in the model's interpretation of the URL plus interaction history. Like Mirage, HTML is the projection layer.
- **Gaps:** Each page is largely independent; not really a stateful application — closer to "creative writing rendered as web pages."

## 3. Adjacent work — clearly *not* projection-based but often confused with it

### 3.1 Generative UI / agent-UI protocols (2024–2026)

These are the technologies the user explicitly asked to distinguish from. All of them are **code-mediated**: the model emits structured data; *deterministic client code* renders it. The model never holds the app's state authoritatively.

- **AG-UI** — https://docs.ag-ui.com/introduction. Event-based protocol (RUN_STARTED, STEP_FINISHED, etc.). Streams state patches between agent backend and frontend. *Frontend renders; backend agent thinks.* Backed by CopilotKit, LangGraph, CrewAI.
- **A2UI** (Google, 2025) — https://a2ui.org/, https://developers.googleblog.com/introducing-a2ui-an-open-project-for-agent-driven-interfaces/. Declarative JSON over A2A or AG-UI. Agent picks from a *catalog of pre-approved components* (Card, Button, TextField). Explicit anti-projection design: "Running arbitrary code generated by an LLM may present a security risk. A2UI is a declarative data format, not executable code."
- **MCP UI / MCP Apps** (Anthropic, official extension Jan 2026) — https://blog.modelcontextprotocol.io/posts/2026-01-26-mcp-apps/, https://mcpui.dev/. Tool calls return UI resources rendered in sandboxed iframes. State lives in the MCP server (= traditional code).
- **Thesys C1** — https://www.thesys.dev/, https://www.thesys.dev/blogs/generative-ui-architecture. "LLM API that turns LLM responses into live, adaptive user interfaces." Outputs structured UI components rendered by the Crayon React SDK. ~300 teams using.
- **Vercel AI SDK Generative UI**, LangGraph generative UI, Flutter GenUI SDK, Google "Generative UI" research blog — all the same shape: model emits structured component data; deterministic renderers display it. https://ai-sdk.dev/docs/ai-sdk-ui/generative-user-interfaces, https://research.google/blog/generative-ui-a-rich-custom-visual-interactive-user-experience-for-any-prompt/
- **Claude Artifacts** — https://support.claude.com/en/articles/11649438. The model writes React/HTML, the *Artifact runtime* executes it. Anthropic explicitly notes "no persistent state… When you close the conversation, the artifact is gone." This is closer to vibe-coding than projection.
- **GenerativeGUI (CHI 2025)** — https://dl.acm.org/doi/10.1145/3706599.3719743. Academic version of Mirage's pattern: LLM generates HTML each conversation turn. Notes that "state management of GUI elements in a chat history is non-trivial" — exactly the gap projection-based apps would need to close.

**Critical analysis:** Every single one of these protocols is built on the assumption that *the agent thinks and code renders*. The model is a generator of UI specifications, not a stateful runtime. They are the engineering counter-thesis to projection: "Don't trust the model with state; serialize state into well-typed events and let real software hold it." A projection-based runtime would treat all of these protocols as scaffolding to be removed once models are reliable enough.

### 3.2 AI app builders (Lovable, Bolt, v0, Replit, Base44, Softr, Manus, Cursor, etc.)

- **Sources:** https://lovable.dev, https://bolt.new, https://v0.dev, https://replit.com, https://base44.com/, https://www.glideapps.com/, https://manus.im
- **Pattern:** User prompts → model generates a full conventional web app (React/Tailwind/Next.js/Postgres). The app then runs as ordinary code; the model is gone at runtime.
- **Relation to projection:** This is *the opposite* of the projection thesis. The model writes code; the code is the app. The UI/state machine lives in the generated source, which is checked in, deployable, debuggable, and entirely model-independent at runtime. Karpathy himself called this "promptable scaffolding — disposable, easily rewritten by AI."
- **Gap:** The whole point of projection-based apps is that this scaffolding *disappears*. App builders are an intermediate step — they accept that we still need code today.

### 3.3 LLM agent frameworks with state (Letta/MemGPT, AIOS, Stately Agent, Manus)

- **MemGPT/Letta** — https://www.letta.com/blog/memgpt-and-letta. Self-editing memory. "Agents move data between in-context core memory (analogous to RAM) and externally stored archival and recall memory (analogous to disk)." Letta paper frames it as "LLM-as-an-Operating-System."
- **AIOS** — https://arxiv.org/abs/2403.16971. LLM kernel with system-call interface, scheduler, context manager, memory manager.
- **Stately Agent** — https://github.com/statelyai/agent. State machines (XState) wrapping LLM agents.
- **Cloudflare Agent Memory** (2026) — https://blog.cloudflare.com/introducing-agent-memory/. "Agent Memory preserves knowledge on compaction instead of losing it."
- **Relation to projection:** These systems acknowledge the *need* for state but solve it by externalizing — explicit memory blocks, vector stores, archival files. The model is *not* trusted to be the canonical state holder; it's a CPU that reads/writes to external memory.
- **Gap:** Projection-based apps would invert this: the model's KV cache *is* the working memory and the model itself is also the memory manager. Letta etc. are pragmatic compromises around the fact that today's models can't be trusted with state.

### 3.4 LLM as OS / "agent OS" research

- **AIOS Compiler** — https://arxiv.org/abs/2405.06907. LLM as interpreter for natural-language programs.
- **AgentOS preprint (Sep 2025)** — https://www.preprints.org/manuscript/202509.0077. Layered architecture: kernel, services, agent runtime, orchestration, user.
- **Qualixar OS** — https://huggingface.co/papers/2604.06392. "Universal OS for AI agent orchestration."
- **AgentOS: From Application Silos to a Natural Language-Driven Data Ecosystem** — https://arxiv.org/html/2603.08938v1.
- **Karpathy's LLM OS thread** — same shape: model = kernel; tools, RAG, code interpreter as syscalls.
- **Relation to projection:** All of these treat the LLM as a *scheduler/orchestrator over external systems*. The "application" is still the union of tools, files, and services; the LLM is the control plane. None propose that the application's data and logic *live in the network's parameters or activations*.
- **Gap:** They're agent-OS designs, not neural-runtime designs. An agent OS still has applications below it; a neural runtime would have only the model and the projection.

### 3.5 Inworld AI (game NPCs)

- **Source:** https://inworld.ai. NVIDIA blog: https://blogs.nvidia.com/blog/generative-ai-npcs/
- **Summary:** "Character Brain" + "Contextual Mesh" + "Real-Time AI." NPC personality, memory, goals modeled as a layered system. Mutations let game devs change personality at runtime.
- **Relation to projection:** Within a single character, this is closer to projection than most agent frameworks — the character *is* a model, and gameplay is interaction with that model. Inworld talks about characters that "learn from context and adjust tone, knowledge, and strategy over time."
- **Gap:** State is still externalized into the Contextual Mesh; the model is one piece of a larger character runtime. And the "app" (the game) is a conventional engine — Inworld plugs into Unreal/Unity. Projection-shaped at the character level, not at the application level.

## 4. Academic literature

### 4.1 Neural program synthesis / program induction

- **Latent Execution for Neural Program Synthesis (LaSynth, NeurIPS 2021)** — Xinyun Chen et al. https://arxiv.org/abs/2107.00101. "Learns a latent representation to approximate the execution of partially generated programs." Decoder takes a latent vector representing the partial program and emits the next vector. The execution itself is learned end-to-end — the model *is* the interpreter.
- **Toward Neural-Network-Guided Program Synthesis and Verification** — https://arxiv.org/abs/2103.09414. Demonstrates extracting integer-arithmetic *programs* directly from network weights.
- **Gradient-Based Program Synthesis with Neurally Interpreted Languages** — https://arxiv.org/html/2604.18907. Model "learns its own symbolic representation directly from data, simultaneously learning a domain-specific neural language and a neural interpreter for such a language."
- **Learning to Synthesize Programs as Interpretable and Generalizable Policies** (NeurIPS 2021) — https://proceedings.neurips.cc/paper_files/paper/2021/file/d37124c4c79f357cb02c655671a432fa-Paper.pdf
- **Relation to projection:** This is the **academic ancestor** of the projection thesis. LaSynth in particular shows that a neural net can hold an *executable* program in its latent space and roll it forward without ever materializing source code. The leap projection-based apps require is from "the program is in the latents" to "the *running* application is in the latents."
- **Gap:** All of these papers target small DSLs and produce programs as *output artifacts*. They demonstrate that a model can *be* an interpreter for a small program — they don't yet let you run an entire user-facing application as a sustained latent state.

### 4.2 World models as runtimes

- **Ha & Schmidhuber, "World Models" (2018)** — https://worldmodels.github.io/. Original work; agent learns a compressed spatial-temporal model of an RL environment and can be trained entirely *inside* that learned world.
- **JEPA family (LeCun, 2022–2025)** — I-JEPA, V-JEPA, LeWorldModel. https://ai.meta.com/blog/yann-lecun-ai-model-i-jepa/, https://ai.meta.com/blog/v-jepa-yann-lecun-ai-model-video-joint-embedding-predictive-architecture/, https://arxiv.org/abs/2603.19312. Predictive architectures over learned latents rather than pixels — "the system creates abstract, high-level representations of input data and predicts future states."
- **Survey: "Is Sora a World Simulator?"** — https://arxiv.org/html/2405.03520v1
- **Survey: "Understanding World or Predicting Future?"** (ACM CSUR 2025) — https://github.com/tsinghua-fib-lab/World-Model
- **Critiques of World Models** — https://arxiv.org/html/2507.05169v1
- **Robotic World Model (Legged Robotics)** — https://github.com/leggedrobotics/robotic_world_model
- **Neural Motion Simulator (MoSim)** — https://arxiv.org/html/2504.07095
- **Relation to projection:** World models are *the* most rigorous formalization of "the app lives in the network." The 2018 Ha/Schmidhuber paper literally trains an agent inside a learned model — i.e., the world *is* the network; observations are projections. This is the projection thesis applied to RL environments.
- **Gap:** World models predict observations, not user-interface states. There is no published work treating, e.g., "an email inbox," "a CRM," "a Linear-style ticket tracker" as a world to be modeled. The semantic gap between "predict the next 720p frame" and "maintain a ticket queue with workflow rules" is enormous.

### 4.3 Differentiable programming / neural runtimes

- **"The Elements of Differentiable Programming"** — https://arxiv.org/abs/2403.14606. Comprehensive treatment of differentiable programming as a paradigm.
- **DiffTaichi, JAX, PyTorch as runtimes.** https://docs.taichi-lang.org/docs/differentiable_programming
- **Pontryagin Differentiable Programming** — https://proceedings.neurips.cc/paper/2020/hash/5a7b238ba0f6502e5d6be14424b20ded-Abstract.html. End-to-end learning of dynamics, policies, and control objectives.
- **Tesseract / pipeline-level differentiable programming** — https://proceedings.scipy.org/articles/kvfm5762
- **"Software 2.0: Neural Codebases and End-to-End Differentiable…"** — https://aijcst.org/index.php/aijcst/article/download/60/53/104
- **Relation to projection:** Differentiable programming establishes that **a runtime can itself be the thing you optimize**. Combined with world models, this is the technical vocabulary needed to describe "the application as a learned program with a learned UI." But these papers focus on physics simulation and control, not on app-shaped problems.
- **Gap:** No one has published a "differentiable application runtime." It is the obvious thing this concept needs.

### 4.4 LLMs as latent-state machines

- **Latent State Estimation Helps UI Agents to Reason** — https://arxiv.org/html/2405.11120v1. Frames zero-shot prompting as forming "point estimates of latent state in a textual space."
- **Transformers are Multi-State RNNs** — https://arxiv.org/html/2401.06104v1. "Decoder transformers can be redefined as a form of multi-state RNNs with an infinite multi-state size."
- **NeuronautLLM** — https://www.sciencedirect.com/science/article/pii/S1524070324000262. Visual analytics of neuron activation; novel projection of influential neurons by activation + semantic explanation.
- **Latent State Models of Training Dynamics** — https://openreview.net/forum?id=NE2xXWo0LF
- **Lightweight Neural App Control (LiMAC)** — https://arxiv.org/html/2410.17883v1. "Hidden states of the transformer corresponding to the UI element embeddings are projected into the same embedding space."
- **Relation to projection:** The Transformers-are-MSRNNs paper is the theoretical license for "use the KV cache as app state." LiMAC is small but interesting — it explicitly *projects* hidden states into a UI-element embedding space, which is structurally what a projection-based app would need.
- **Gap:** None of these go from "the model has latent state" to "the latent state *is* a database with constraints I can query."

## 5. Concepts that overlap conceptually but diverge in execution

### 5.1 "Vibe coding" / Karpathy MenuGen
- https://karpathy.bearblog.dev/vibe-coding-menugen/, https://en.wikipedia.org/wiki/Vibe_coding
- Vibe coding = prompt → code → app. Still a code layer. Karpathy's later "MenuGen would have been better as just-in-time Gemini behavior" framing is the projection critique of vibe coding.

### 5.2 RAG / agent memory / context engineering
- Anthropic's "Effective context engineering" (2026), Manus's Context Engineering blog, SwirlAI 2026 state-of-the-art roundup. https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents, https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus
- These are the engineering response to "the model can't hold real state": shape what's in its window, retrieve, compress, route. Pragmatic *negation* of the projection thesis: state lives outside; we just feed the model views.

### 5.3 LLM-as-database
- "The Database Has a New User—LLMs" (Tiger Data), "Stop Treating Your LLM Like a Database" (Sean Falconer). https://www.tigerdata.com/blog/the-database-new-user-llms-need-a-different-database, https://seanfalconer.medium.com/stop-treating-your-llm-like-a-database-87e8c89a0475
- Conventional wisdom: don't. The model is not a reliable store. Projection-based apps inherit this objection directly — they require either (a) much more reliable state representation in models, or (b) a hybrid where state lives in latents but is checkpointed/audited.

### 5.4 Living documents (Elicit)
- https://blog.elicit.com/living-documents-ai-ux/. Spreadsheet whose cells are populated by thousands of LLM calls, with the spreadsheet *itself* as the user-visible artifact.
- Closer to projection in spirit: the "app" is the document, and the LLM continually updates it. But state lives in the document (regular code), not in the model.

## 6. The gap analysis — where projection-based apps actually require new work

Bringing this all together, the path from current systems to true projection-based apps has four open problems, none of which has a clean published solution:

| Open problem | What current work gets right | What is missing |
|---|---|---|
| **Persistent state in latents** | World models (Genie 3, Oasis) maintain coherent state for ~minutes. MemGPT/Letta externalize state into well-typed memory blocks. | No model holds business-application state (users, records, workflow positions) reliably for hours/days inside its weights or KV cache. Genie 3's ~1 minute consistency is the SOTA. |
| **Projection from state to UI** | A2UI, AG-UI, Mirage, Thesys C1 all show ways to get UI from model output. | None of them treat the UI as a *learned function of hidden state*. They go state → tokens → UI-spec → render. Projection-based apps need state → UI directly (potentially via a small learned decoder over activations). |
| **Verifiability / auditing** | Letta's memory blocks, AG-UI's typed events, MCP's JSON-RPC are all auditable. | A model whose hidden state *is* the app is a black box. No work on "auditing the running state of a neural app" exists; it's mostly mechanistic-interpretability adjacent. |
| **Interaction latency** | GameNGen and Oasis hit 20 fps for video. | Mirage is seconds-per-click. There is no latency-acceptable projection-based productivity app today. |

## 8. Honest critical take

The projection-based-application concept, as defined in §0, **does not yet exist as a deployed system or a published research artifact** outside of narrow demos (Mirage, GameNGen, Genie 3, Oasis). Every piece of vocabulary it would use exists in the literature, scattered:

- "Weights are code" (Karpathy, Software 2.0)
- "Network is the runtime" (GameNGen, Oasis, Genie 3)
- "Latent state holds the program" (LaSynth)
- "KV cache is RAM" (Letta, Transformers-as-MSRNNs)
- "UI is a projection of model output" (A2UI, generative UI, GenerativeGUI)

But the synthesis — a system where one model holds application state, business logic, and UI rendering all at once, with the visible UI being a learned projection — is **not** any of:

- Generative UI (model outputs UI specs, code renders)
- Agent runtimes (model orchestrates external state)
- World models (no app/UI layer, no business semantics)
- AI app builders (model writes code; code is the app)
- Vibe coding (same)
- LLM-as-OS proposals (model is a kernel over external resources)

The closest published artifact is **Mirage** — and its author calls it a toy. The closest framing is **Karpathy's MenuGen anecdote** — and he stops short of advocating for it as an architecture. The closest research lineage is **world models + LaSynth-style latent execution** — and neither has been pushed into the application domain.

Anyone building toward projection-based apps is therefore standing on (a) a thin demo (Mirage), (b) an untested architectural claim ("the network *can* be the application"), and (c) a series of negative results from the protocol/agent space showing that the field, when forced to ship, externalizes state every time. That is either a moat or a warning sign, depending on whether you think the underlying model capability is the bottleneck or whether the externalization is fundamental.

The strongest argument *for* projection-based apps in the current literature is implicit: world models and neural game engines work. If a network can be DOOM, in principle a network can be a CRM. The strongest argument *against* is that DOOM has no business invariants ("ammo count must equal the sum of pickups minus shots fired"), and CRMs do — and we have no published technique for guaranteeing such invariants inside a network's hidden state.

The interesting research bet would be: **specify the projection-based app architecture formally enough that it can be falsified.** No one has yet.
